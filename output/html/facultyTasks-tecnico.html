Ciclo de vida del WebPart

Responsabilidades:

  1. Lee this.context.pageContext.user.loginName, cultura, web URL, MSGraphClientFactory, SPHttpClient.
  2. Evalúa this.context.sdks.microsoftTeams para detectar contexto Teams.
  3. Inicializa Teams SDK si está disponible: microsoftTeams.app.initialize().
  4. Persiste el serviceScope completo en useAppStore para consumo de servicios desde cualquier punto.

Impacto: Sin onInit correcto, los servicios no pueden acceder a las dependencias SPFx y la app falla.

Monta el árbol React completo en this.domElement usando ReactDom.render(). El componente raíz es FacultyTasks, que recibe todo el estado desde los stores Zustand.

Recibe el objeto de tema del host (SharePoint o Teams) y propaga las variables CSS semánticas sobre el domElement. Garantiza que Fluent UI adapte colores automáticamente.

onDispose: Desmonta React limpiamente con ReactDom.unmountComponentAtNode().

getPropertyPaneConfiguration: En versión actual no expone campos funcionales. Configuración se realiza vía appSettings compilado.

Modelo de tipos y DTOs

Tipos internos (dominio)

  • Period: Período académico con fechas convertidas a Date, parámetros de planificación, array de festivos
  • Availability: Agregación de disponibilidades de usuario por período. Contiene array de fechas con status (AvailabilityStatus enum) y número de semana ISO
  • Assignment: Asignación docente con status (pending/accepted/rejected), compatibilityStatus (compatible/conflicted/unknown) y datos completos de la asignación
  • Task: Tarea operativa con tipo, descripción, estado (0=pending, 1=completed, 2=expired), URL, fecha límite, supervisor e imageUrl (enriquecida progresivamente)

DTOs del backend

Representan exactamente el contrato JSON devuelto por el backend. Diferencias clave respecto a tipos internos:

  • Fechas en DTOs: strings ISO → convertidas a Date en tipos internos
  • Status numéricos en DTOs → mapeados a enums de string en tipos internos
  • PeriodDTO incluye hoursPerDay y gapsCoefficient; UserPeriodDTO para profesor no necesita estos campos

Riesgos técnicos identificados

Descripción: Ambos roles comparten GUID en appSettings.

Impacto: Imposibilidad de tener usuarios solo con rol taskUser sin también tener rol user.

Mitigación: Revisar con administrador del tenant si la separación de grupos es necesaria. Si sí, crear grupo distinto y actualizar GUID en appSettings.

Descripción: WebPart no declara hosts objetivo en el manifest.

Impacto: Posibles problemas de visibilidad o publicación en entornos específicos.

Mitigación: Declarar explícitamente ["SharePointWebPart", "TeamsPersonalApp", "TeamsTab"] según entorno objetivo.

Descripción: Servicios devuelven undefined o false en error; sin componente genérico de error.

Impacto: Usuario puede ver pantallas vacías sin explicación si API falla.

Mitigación: Implementar React Error Boundary y mensajes de feedback al usuario.

Descripción: DTOs del frontend son inferidos del consumo observado.

Impacto: Si backend cambia un campo, frontend puede fallar silenciosamente.

Mitigación: Generar y versionar contrato OpenAPI del backend, usarlo como fuente de tipos.

Descripción: Llamadas al backend no implementan reintentos en error transitorio.

Impacto: Error de red puntual puede dejar app en estado de carga fallida.

Mitigación: Implementar reintentos con backoff exponencial para llamadas GET no críticas.

Descripción: Paths para MySessions, Requests, Evaluations definidos en Constants pero sin componentes activos.

Impacto: Deep links a estas rutas no funcionarán hasta activación.

Mitigación: Eliminar constantes de rutas no activas u añadir componente placeholder que informe al usuario.
